The App's Front Door

Every Flutter app starts with a main.dart file. It's the front door — the first thing that runs when you open the app. Here's how ScoreCaddie sets up the stage:

CODE 
void main() async {
  WidgetsFlutterBinding.ensureInitialized();
  await dotenv.load(fileName: '.env');

  if (dotenv.env['ONESIGNAL_APP_ID'] != null) {
    OneSignal.initialize(dotenv.env['ONESIGNAL_APP_ID']!);
  }

  await Supabase.initialize(
    url: AppConfig.supabaseUrl,
    anonKey: AppConfig.supabaseAnonKey,
    authOptions: const FlutterAuthClientOptions(
      authFlowType: AuthFlowType.pkce,
    ),
  );

  final db = AppDatabase();
  await seedCourses(db);

  runApp(
    ProviderScope(
      overrides: [databaseProvider.overrideWithValue(db)],
      child: const ScoreCaddieApp(),
    ),
  );
}
PLAIN ENGLISH

This is the main function — the first thing that runs when the app opens.

Make sure Flutter is ready before doing anything else.

Load the secret keys from the .env file (API keys, etc.).

If there's a OneSignal ID, set up push notifications.

Connect to the OneSignal service with that ID.

End of the push notification setup.

Connect to Supabase (the cloud backend) using the URL and key from the config.

Use PKCE for secure authentication — a fancy way of saying "make login secure."

Create the local SQLite database (Drift).

Fill the database with the list of Kenyan golf courses.

Start the app, wrapping it in a ProviderScope so Riverpod can manage state.

Pass the database into the provider system so every screen can access it.

Launch the actual app widget — the ScoreCaddieApp.

The Screen Scaling Trick

Here's a clever pattern in app.dart most developers miss. The app uses a design baseline to scale the entire UI on smaller phones:

CODE 
const double baselineWidth = 390.0;

if (width < baselineWidth && width > 0) {
  final double scale = width / baselineWidth;
  return FittedBox(
    fit: BoxFit.fitWidth,
    alignment: Alignment.topCenter,
    child: SizedBox(
      width: baselineWidth,
      height: height / scale,
      child: child,
    ),
  );
}
PLAIN ENGLISH

The UI was designed for a 390-pixel-wide screen (iPhone 14 size).

If the phone is narrower than that...

Calculate how much smaller the screen is (e.g., 0.85 for an older iPhone).

Wrap the entire app in a FittedBox — like putting the UI in a shrinkable bag.

Shrink the bag to fit the screen width.

Keep the bag anchored at the top so nothing gets cut off.

Inside the bag, the UI still thinks it's 390px wide.

But the height gets adjusted so nothing gets squished.

Put the actual app content inside this scaled container.

If the screen is 390px or wider, just show the app normally.

💡
Key Insight

This is a fixed-baseline scaling approach. Instead of rewriting layouts for every screen size, the app assumes one size and scales down. It's faster to build but less flexible than true responsive design. If you asked an AI to "make this responsive," it might replace this with a more flexible system.
