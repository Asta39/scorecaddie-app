  WidgetsFlutterBinding.ensureInitialized();
  await dotenv.load(fileName: '.env');

  if (dotenv.env['ONESIGNAL_APP_ID'] != null) {

  await seedCourses(db);

  runApp(
    ProviderScope(
      overrides: [databaseProvider.overrideWithValue(db)],

    ),
  );
}

The Screen Scaling Trick

Here's a clever pattern in app.dart most developers miss. The app uses a design baseline to scale the entire UI on smaller phones:

CODE 
const double baselineWidth = 390.0;

if (width < baselineWidth && width > 0) {
  final double scale = width / baselineWidth;
  return FittedBox(
    fit: BoxFit.fitWidth,
    alignment: Alignment.topCenter,
    child: SizedBox(
      width: baselineWidth,
      height: height / scale,
      child: child,
    ),
  );
}
PLAIN ENGLISH

The UI was designed for a 390-pixel-wide screen (iPhone 14 size).

If the phone is narrower than that...

Calculate how much smaller the screen is (e.g., 0.85 for an older iPhone).

Wrap the entire app in a FittedBox — like putting the UI in a shrinkable bag.

Shrink the bag to fit the screen width.

Keep the bag anchored at the top so nothing gets cut off.

Inside the bag, the UI still thinks it's 390px wide.

But the height gets adjusted so nothing gets squished.

Put the actual app content inside this scaled container.

If the screen is 390px or wider, just show the app normally.

💡
Key Insight

This is a fixed-baseline scaling approach. Instead of rewriting layouts for every screen size, the app assumes one size and scales down. It's faster to build but less flexible than true responsive design. If you asked an AI to "make this responsive," it might replace this with a more flexible system.

You're on the 14th hole at Muthaiga Golf Club. The trees are thick, the clubhouse is far, and your phone shows one bar of 3G. You just birdied the hole and want to record your score. If the app needed the internet to save, you'd be stuck.

ScoreCaddie solves this with an offline-first design. Every score, every round, every practice session is saved to your phone's SQLite database instantly. The cloud is just a backup that catches up when it can.

The Stable UUID Trick

Here's a problem: you save a round on your phone, then later the app tries to sync it to the cloud. But what if the round already exists in the cloud from another device? You'd get duplicates. The solution is deterministic UUIDs:

CODE 
// Stable namespace for deterministic UUIDs
static const _namespace = '6ba7b810-9dad-11d1-80b4-00c04fd430c8';

String _generateStableId(String type, List<String> parts) {
  final input = '$type:${parts.join(':')}';
  return _uuid.v5(_namespace, input);
}
PLAIN ENGLISH

This is a fixed "seed" string that acts like a secret recipe key.

This function takes a type (like "round" or "shot") and a list of parts.

Combine them into one string like "round:123:2024-06-01".

Run it through UUIDv5 — a hash function that always produces the same ID for the same input.

The result: the same round always gets the same ID, no matter which device creates it.

💡
Key Insight

This pattern is like a library's card catalog — instead of randomly generating a new card number every time, the catalog uses the book's title and author to create a consistent number. Same book, same number. No duplicates.

Profile Self-Healing

Here's another clever pattern. When a user logs in, the app checks if their profile exists in the cloud database. If not, it creates a minimal one automatically. If the profile says "Golfer" but Google has their real name, it updates it. The app fixes itself:

CODE 
// SELF-HEALING: Ensure user profile exists
final dbProfile = await _client.from('User')
  .select('name, role, bio')
  .eq('id', user.id)
  .maybeSingle();

if (dbProfile == null) {
  await _client.from('User').insert({
    'id': user.id,
    'email': user.email ?? '',
    'name': user.metadata?['full_name'] ?? 'Golfer',
    'role': currentRole,
  });
} else if (dbProfile['name'] == 'Golfer'
    && user.metadata?['full_name'] != null) {
  await _client.from('User').update({
    'name': user.metadata?['full_name'],
  }).eq('id', user.id);
}
PLAIN ENGLISH

Look for the user's profile in the cloud database.

Grab the name, role, and bio fields for this user ID.

If nothing comes back...

Create a brand new profile with their ID and email.

Use their Google name if available, otherwise default to "Golfer".

Set their role (player, coach, or admin).

If the profile exists but says "Golfer" and Google has a real name...

The Code Behind Daniel

Here's the actual code that makes Daniel work. The getCoachingFeedback method sends your shot data to Groq and returns text advice:

CODE 
static Future<String> getCoachingFeedback({
  required Map<String, dynamic> shot,
  required db.UserProfile player,
  required List<Map<String, dynamic>> recentShots,
}) async {
  final response = await http.post(
    Uri.parse(_groqChatUrl),
    headers: {
      'Authorization': 'Bearer $_groqApiKey',
      'Content-Type': 'application/json',
    },
    body: jsonEncode({
      'model': _groqModel,
      'temperature': 0.7,
      'max_tokens': 150,
      'messages': [
        {'role': 'system', 'content': _buildSystemPrompt(player)},
        {'role': 'user', 'content': 'CURRENT SHOT: $shot. RECENT: $recentShots'},
      ],
    }),
  ).timeout(const Duration(seconds: 12));

  if (response.statusCode != 200)
    throw Exception('Daniel is quiet right now.');

  final data = jsonDecode(response.body);
  return data['choices'][0]['message']['content'];
}

The current shot data (club, distance, direction, etc.).

Your profile so Daniel knows your name and handicap.

Your recent shots so Daniel can spot patterns.

Send a POST request to the Groq API endpoint.

Temperature 0.7 means "be creative but not random."

Limit the response to 150 tokens (about 2-3 sentences).

First message: the system prompt that tells Daniel who he is.

Second message: the actual shot data Daniel needs to analyze.

If the API returned an error, throw a friendly exception.

The Handicap Engine

How ScoreCaddie calculates your official golf handicap

What Is a Handicap, Really?

A handicap is golf's great equalizer. It's a single number that lets a 28-handicap beginner play against a 5-handicap skilled player — and both have a chance to win. The lower your handicap, the better you are.

ScoreCaddie implements the official WHS 2024 formulas, which means your handicap is calculated the same way as at any professional golf club. Here's the pipeline:

1
ESC Cap

Adjust any blow-up holes to a max of Net Double Bogey

2
Score Differential

Convert your adjusted score to a standardized number

3
Best 8 of 20

Take the best 8 differentials from your last 20 rounds

4
Caps & ESR

Apply soft/hard caps and exceptional score reductions

The Score Differential Formula

Here's the core math that turns a round of golf into a handicap-relevant number. The formula is:

(Adjusted Score - Course Rating) × (113 / Slope Rating)

What each piece means:

Adjusted Score Your actual score, but any hole worse than Net Double Bogey is capped
Course Rating What a scratch golfer (0 handicap) would score on this course
Slope Rating How much harder the course is for average golfers (113 is average)
113 The standard slope rating — a constant in the formula
💡
Key Insight

The slope rating is why your handicap travels with you. If you shoot 85 at an easy course (slope 110), that's good. If you shoot 85 at a hard course (slope 135), that's excellent. The formula adjusts for difficulty so your handicap is fair no matter where you play.

The Code That Calculates Your Handicap

Here's the actual calculateHandicapIndex method from the WHSEngine. It takes your last 20 differentials and produces a single handicap number:

CODE 
static double? calculateHandicapIndex(
  List<double> allDifferentials,
  {double? lowIndex, double? latestScoreDiff, double? previousIndex}
) {
  if (allDifferentials.isEmpty) return null;

  final recent = allDifferentials.length > 20
      ? allDifferentials.sublist(allDifferentials.length - 20)
      : allDifferentials;

  final count = recent.length;
  final numToUse = _getDifferentialsToUse(count);
  if (numToUse == 0) return null;

  final sorted = List<double>.from(recent)..sort();
  final best = sorted.sublist(0, numToUse);
  final average = best.reduce((a, b) => a + b) / best.length;

  double result = average * 0.96;

  if (latestScoreDiff != null && previousIndex != null)
    result += calculateExceptionalScoreReduction(latestScoreDiff, previousIndex);

  if (lowIndex != null)
    result = applyYearlyCap(result, lowIndex);

  return (result * 10).roundToDouble() / 10;
}
PLAIN ENGLISH

Takes a list of all your past score differentials.

The list of all your past differentials.

Optional: your lowest index this year and your latest differential.

If you have no scores, return nothing (can't calculate a handicap).

Only use the most recent 20 scores. WHS looks at a rolling window.

If more than 20, keep only the last 20.

Otherwise use all of them.

Count how many differentials we have.

Look up how many to use based on the count (1 for 3-5 rounds, 2 for 6-8, etc.).

If none to use, return nothing.

Sort the differentials from lowest (best) to highest (worst).

Take only the best N differentials (the low ones).

Calculate the average of those best differentials.

Multiply by 0.96 — the WHS averaging factor.

If we have a latest score and previous index...

Check if the latest score was exceptional and reduce the handicap if so.

If we have a low index anchor...

Apply the soft/hard cap so your handicap can't jump too fast.

Round to one decimal place and return.
